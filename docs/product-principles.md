# Product Principles

## Production first

This project is for real presentations, not training exercises or throwaway demos. Features should be evaluated based on whether they improve a production authoring and presentation workflow.

## Credibility matters

The tool should help authors present with clarity, confidence, and credibility. Output should feel polished, readable, and trustworthy.

## Accessibility is a product requirement

Accessibility applies to both the editor and the exported presentation output. It is not optional and should be enforced through design, validation, and testing.

## Google Slides feel, Markdown source

The experience should feel responsive and practical like a modern slide editor, while keeping Markdown as the machine-readable source of truth.

## Notes support, not dependence

Slides should communicate the main idea clearly. Speaker notes are for detail, prompts, and delivery support, not for hiding essential meaning from the audience.

## Motion discipline

Motion should be conservative by default. Avoid auto-advance, parallax, 3D transitions, and decorative effects that reduce comprehension or accessibility.

## Static by default

The baseline product must work on GitHub Pages without a server. Local browser storage, caching, and portable export are preferred over infrastructure-heavy solutions.

## AI is optional, author control is required

AI can help review and improve decks, especially through local-first tools such as Ollama, but it must remain optional. Suggestions should be reviewable, reversible, and never silently overwrite author work.

## Reusable, but not dependent on adoption

The project should be strong enough that teams such as CivicActions and W3C could reuse it, but success should not depend on that happening.
