## Purpose

This constitution captures the technical standards, code quality expectations,
tribal knowledge, and governance rules for Markdown Slides Editor. All features
and pull requests should align with these principles.

## Technical Standards

### Languages and Frameworks

Plain JavaScript (ES modules, no transpiler), HTML, and CSS. No frontend
framework, no bundler. Node.js 18+ for the test runner. Python 3.9+ for the
local dev server only. The app runs as a static browser application — GitHub
Pages static hosting is a core constraint.

### Testing Requirements

Node built-in test runner (`node --test`) for unit tests. Cucumber.js for BDD
scenarios. Semantic output checks via custom script. No coverage target yet,
but all new features must have tests. `npm test` must pass before finishing any
change.

### Performance and Scale

Not a primary concern. The app is a single-user browser tool. Key constraint:
keep the app shell lightweight enough for offline-first service worker caching.
No server-side performance targets.

### Deployment and Constraints

GitHub Pages static hosting. No server requirement for the default experience.
No build step, no transpiler, no bundler. Must work as plain static files
served over HTTPS. Optional Cloudflare Worker integration planned for
authenticated save/load but must not be required.

## Code Quality

### Pull Request Requirements

Single maintainer project. AI agent PRs (Copilot, Claude) require human review
before merge. Self-merge allowed after tests pass and manual verification for
maintainer-authored changes.

### Code Review Checklist

- Tests pass
- Accessibility not regressed (heading structure, keyboard access, focus states, reduced-motion)
- No server dependency introduced
- AI features remain optional
- Source format contract preserved (front matter, `---`, `Note:`, `Resources:`, `Script:`)
- Browser shortcuts not hijacked

### Quality Gates

- `npm test` must pass (unit + semantic + BDD)
- Manual browser verification required for UI changes (editor, audience view, presenter view)
- No new external runtime dependencies without justification
- Accessibility checklist reviewed for UI changes

### Documentation Standards

- Update `README.md` when behavior changes materially
- Update `AGENTS.md` when project layout or commands change
- Disclose AI use accurately in `README.md`
- Code should be self-documenting — minimal inline comments
- `ACCESSIBILITY.md` and `docs/` updated when accessibility behavior changes

## Tribal Knowledge

### Team Conventions

- Prefer small, direct edits over new abstractions
- Static baseline and optional AI layer are separate — never blur them
- Accessibility is a product requirement, not polish
- Prefer browser-native primitives (IndexedDB, localStorage, service worker) over new infrastructure
- Keep audience view clean — presentation controls belong in presenter view only
- Whisper/AI UI only surfaces when capability is actually available

### Lessons Learned

- htmx was evaluated and rejected for the static baseline (no server to return HTML fragments)
- Keep dependencies minimal — the only devDependency is Cucumber.js
- AI agent PRs need careful human review; agents sometimes introduce unnecessary abstractions or break the source format contract
- Generated presentation HTML must stand alone as valid, semantic markup

### Historical Decisions

- Chose no build step and no framework for maximum static hosting compatibility and simplicity
- Built a custom Markdown renderer instead of adopting a library — only supports the syntax actually needed, keeps the dependency count at zero
- Chose IndexedDB-first persistence with localStorage fallback for offline resilience
- Chose BroadcastChannel for presenter/editor sync with localStorage fallback
- Chose AGPLv3 license
- Planned whisper-slides runtime alignment is tracked but not yet implemented
- Chose WCAG 2.2 AA as the conformance target for both editor and output

## Governance

Any team member can propose amendments via pull request. Changes are discussed
and merged following standard PR review process.

### Compliance Validation

Code reviewers validate compliance during PR review. Constitution violations
should be flagged and addressed before merge.

### Exception Handling

Exceptions discussed case-by-case with team. Strong justification required.
